# 抢位赛（定区升降赛）模块设计文档

> **日期**: 2025-01-24
> **功能**: 赛事管理 — 新增抢位赛/定区升降赛完整支持
> **方案**: 方案一（最小侵入式复用现有模型）

---

## 一、需求概述

根据 `docs/比赛规则/定区升降赛.md` 规则，在现有赛事管理模块中新增"抢位赛"（定区升降赛/Promotion-Relegation）的完整前后端支持。

**核心规则**: 选手占据固定位置（1号位=冠军位），低位选手挑战相邻高位，胜者上升交换位置，败者下降，位置实时变动。

**关键决策（已确认）**:
- 初始位置分配：**抽签随机入位**
- 挑战发起方式：**叫号模式**（管理员手动发起）
- 刚比赛过的选手保护：**不限制**（完全由管理员人工判断）

---

## 二、兼容性保障（重要）

本模块的实现对已有模块**零侵入、零影响**：

| 已有模块 | 保障措施 |
|---------|---------|
| 分组循环赛（ROUND_ROBIN） | 不修改其任何生成/排名逻辑 |
| 纯小组赛（PURE_GROUP） | 不修改其任何生成/排名逻辑 |
| 单败淘汰赛（SINGLE_ELIMINATION） | 不修改 KnockoutService |
| 锦标赛（CHAMPIONSHIP） | 不修改其小组赛+淘汰赛双阶段逻辑 |
| 参赛队员管理 | 仅新增一个 nullable 字段，不影响现有增删改查 |
| 比分录入 | 仅在 `round_type=PROMOTION_RELEGATION` 时追加换位逻辑，其他类型完全跳过 |
| 前端赛事列表页 | 不修改表格结构，仅新增一个标签页（条件渲染） |

## 三、数据模型改动

### 3.1 新增字段

**文件**: `backend/app/plugin/module_badminton/tournament/model.py`

在 `TournamentParticipantModel` 中新增：

```python
current_position: Mapped[int | None] = mapped_column(
    Integer, nullable=True, comment="抢位赛当前位置（1=冠军位，数字越大排名越低）"
)
```

**说明**:
- 该字段为 `nullable=True`，对已有数据无影响
- 仅在 `tournament_type = PROMOTION_RELEGATION` 时有效
- 赛事结束时，`current_position` 的顺序即为最终排名

### 3.2 无需新增表

复用现有模型：
- `TournamentModel` — 赛事基本信息
- `TournamentParticipantModel` — 参赛队员 + 位置
- `TournamentMatchModel` — 比赛记录（`round_type=PROMOTION_RELEGATION`）

## 四、后端 API 设计

### 4.1 新增 API

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| `POST` | `/{tournament_id}/positions/init` | `tournament:create` | 随机抽签初始化位置 |
| `GET` | `/{tournament_id}/positions` | `tournament:list` | 获取当前位置板 |
| `POST` | `/{tournament_id}/challenge` | `tournament:update` | 管理员发起挑战，创建比赛 |
| `GET` | `/{tournament_id}/pr-matches` | `tournament:list` | 获取抢位赛历史比赛 |

### 4.2 复用/修改的 API

| 方法 | 路径 | 改动说明 |
|------|------|---------|
| `POST` | `/{tournament_id}/generate-matches` | 对 PROMOTION_RELEGATION 类型，调用 `init_positions` 而非生成循环赛对阵 |
| `PUT` | `/{tournament_id}/matches/{match_id}/score` | 录入比分后，若 `round_type=PROMOTION_RELEGATION`，追加**自动换位**逻辑 |

### 4.3 API 详细定义

#### POST /{tournament_id}/positions/init

**Request Body**: 无

**业务逻辑**:
1. 校验赛事类型 = PROMOTION_RELEGATION
2. 获取所有未退赛的参赛者
3. 随机打乱顺序
4. 按顺序分配 `current_position = 1, 2, 3, ...`
5. 批量更新数据库

**Response**:
```json
{
  "code": 200,
  "msg": "位置初始化成功",
  "data": [
    {"participant_id": 1, "student_name": "张三", "current_position": 1},
    {"participant_id": 2, "student_name": "李四", "current_position": 2},
    ...
  ]
}
```

#### GET /{tournament_id}/positions

**Response**:
```json
{
  "code": 200,
  "msg": "获取成功",
  "data": [
    {"participant_id": 1, "student_name": "张三", "current_position": 1},
    {"participant_id": 2, "student_name": "李四", "current_position": 2},
    ...
  ]
}
```
按 `current_position` 升序排列。

#### POST /{tournament_id}/challenge

**Request Body**:
```json
{
  "challenger_id": 15,
  "defender_id": 14
}
```

**校验规则**:
1. 两选手必须属于同一赛事
2. 两选手均未退赛
3. `challenger.current_position == defender.current_position + 1`（必须是相邻低位挑战高位）
4. 两选手之间不存在 `status=SCHEDULED/IN_PROGRESS` 且未结束的比赛

**业务逻辑**:
1. 创建 `TournamentMatchModel`，`round_type=PROMOTION_RELEGATION`
2. `player1_id = challenger_id`, `player2_id = defender_id`
3. `status = SCHEDULED`

**Response**: 返回创建好的比赛详情

#### GET /{tournament_id}/pr-matches

**Response**: 返回该赛事下所有 `round_type=PROMOTION_RELEGATION` 的比赛列表，按创建时间倒序。

### 4.4 比分录入的自动换位逻辑

在 `TournamentMatchService.record_score_service` 中，追加以下逻辑（仅在 `round_type=PROMOTION_RELEGATION` 时执行）：

```python
# 1. 判断谁是挑战者，谁是守位者
match = await TournamentMatchCRUD(auth).get(id=match_id)
if match.round_type == RoundTypeEnum.PROMOTION_RELEGATION and winner_id:
    challenger = await TournamentParticipantCRUD(auth).get(id=match.player1_id)
    defender = await TournamentParticipantCRUD(auth).get(id=match.player2_id)
    
    # 2. 如果挑战者获胜，交换位置
    if winner_id == match.player1_id:
        challenger_pos = challenger.current_position
        defender_pos = defender.current_position
        await TournamentParticipantCRUD(auth).update(match.player1_id, {"current_position": defender_pos})
        await TournamentParticipantCRUD(auth).update(match.player2_id, {"current_position": challenger_pos})
    # 3. 守位者获胜，位置不变（无需操作）
```

### 5.4 业务流程交互

```
管理员打开赛事管理 → 点击"抢位赛"标签页
  └─ 如果未抽签 → 显示"随机抽签入位"按钮
       └─ 点击 → 后端随机分配位置 → 刷新显示位置板
  └─ 如果已抽签 → 显示位置板 + 历史比赛记录
       └─ 管理员点击某行"挑战"按钮
            └─ 弹出确认框 → 调用 /challenge → 创建比赛
            └─ 选手比赛结束后 → 管理员在历史记录中点击"录入比分"
                 └─ 弹出 ScoreDialog → 调用 /score
                      └─ 如果挑战者胜 → 后端自动交换位置 → 前端刷新位置板
                      └─ 如果守位者胜 → 位置不变
```

## 七、安全与校验

| 校验点 | 处理方式 |
|--------|---------|
| 非抢位赛赛事调用抢位赛 API | 返回 400，提示"仅定区升降赛支持此操作" |
| 挑战非相邻位置 | 返回 400，提示"只能挑战相邻高位的选手" |
| 重复发起未完成的挑战 | 返回 400，提示"两选手之间有未完成的比赛" |
| 已退赛选手被挑战 | 返回 400，提示"选手已退赛" |
| 非草稿状态修改位置 | 允许，因为抢位赛的比赛是在 ACTIVE 状态下动态进行的 |
